### Генератор спроса
Мультипликативная модель вместо шума вокруг константы: base × сезон года × день недели × погода × событие × шум.
Календарь: тик = день, эпизод стартует со случайного дня года. Погода — та же `ambient_weather`, что влияет на порчу.
События-пики (фестиваль/промо, 1–3 дня, ×1.5–2.5) обязательны — на них трансформер должен обыгрывать naive-бейзлайны.

### Датасет для трансформера
Спрос экзогенный — генератор крутится напрямую, без агентов, той же функцией из `core`.
~50 рядов × 2000 дней, колонки: ковариаты + demand. Сплит train/val/test по времени, не вперемешку.
Учим на чистом demand; sold (цензурированный запасом) — потом, отдельным апгрейдом вместе с реальным датасетом.

### Трансформер
Окно 28 дней [спрос + ковариаты] → спрос следующего дня. Маленький: 2–3 слоя, d_model 64.
Скрипт в `training/`, чекпоинт в `artifacts/modules/forecaster/`.

### Оценка прогноза
MAE/MAPE против last value и seasonal naive + сравнение с теоретическим потолком (noise floor генератора известен — посчитать «% выжатого предсказуемого»). Отдельный срез по дням событий.

### Буфер истории и прогноз в среду
Среда хранит скользящее окно продаж и ковариат (sin/cos дня года и недели, погода, без флага будущих событий — пики форкастер замечает по истории продаж); на каждом шаге форкастер заполняет `state.demand_forecast`. Ковариаты — вход только форкастера: obs агентов не трогаем, inventory видит ровно 4 поля Алг. 4, сезонность доходит через `state.demand_forecast`. Для delivery по Алг. 5 прогноз F задаёт генерацию спроса эпизода.

### Банк сценариев сбоев
Подключить LLM для генерации JSON-сценариев сбоев (disruption engine). Scenario parser маппит сценарии строго в существующие структурные переменные (по статье: blocked nodes / increased transit / risk flags): закрытый маршрут → `BLOCKED_NODE`, задержка → `INCREASED_TRANSIT`, отказ холодильника → `RISK_FLAG`/`fault_signals`, скачок спроса → event-механизм генератора (`event_multiplier`), не новый тип disruption.
